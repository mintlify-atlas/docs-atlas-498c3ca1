## Overview

Better Fetch is built with TypeScript-first design, providing full type safety for requests, responses, and errors. This ensures compile-time guarantees and excellent IDE support for your API calls.

## Basic Type Inference

Specify response types using TypeScript generics:

```typescript
interface User {
  id: string;
  name: string;
  email: string;
  createdAt: string;
}

const { data, error } = await betterFetch<User>(
  "https://api.example.com/users/123"
);

if (data) {
  // TypeScript knows data is of type User
  console.log(data.name);  // ✓ Type-safe
  console.log(data.age);   // ✗ Error: Property 'age' does not exist
}
```

## Response Type Generics

The `betterFetch` function accepts up to three type parameters:

```typescript
betterFetch<TRes, TErr, Options>(
  url: string,
  options?: Options
)
```

<Steps>
  <Step title="Response Type (TRes)">
    The type of the successful response data:
    
    ```typescript
    interface Post {
      id: number;
      title: string;
      body: string;
    }
    
    const { data, error } = await betterFetch<Post>(
      "https://api.example.com/posts/1"
    );
    // data is Post | null
    ```
  </Step>
  
  <Step title="Error Type (TErr)">
    The type of the error response:
    
    ```typescript
    interface ApiError {
      code: string;
      message: string;
      details?: Record<string, string[]>;
    }
    
    const { data, error } = await betterFetch<Post, ApiError>(url);
    
    if (error) {
      // TypeScript knows about error.code and error.message
      console.log(error.code);
      console.log(error.status);     // Always available
      console.log(error.statusText); // Always available
    }
    ```
  </Step>
  
  <Step title="Options Type">
    The type of the options object (usually inferred):
    
    ```typescript
    const options = {
      method: "POST" as const,
      body: { title: "New Post" }
    } satisfies BetterFetchOption;
    
    const { data } = await betterFetch<Post>(url, options);
    ```
  </Step>
</Steps>

## Type-Safe Request Options

### Headers

Better Fetch provides type-safe headers with autocomplete for common headers:

```typescript
const { data } = await betterFetch("https://api.example.com/data", {
  headers: {
    "accept": "application/json",           // ✓ Autocomplete available
    "content-type": "application/json",     // ✓ Autocomplete available
    "authorization": "Bearer token",        // ✓ Autocomplete available
    "x-custom-header": "custom-value"      // ✓ Any string allowed
  }
});
```

You can extend the headers type for custom headers:

```typescript
interface CustomHeaders {
  "x-tenant-id": string;
  "x-api-version": string;
}

const { data } = await betterFetch<User, unknown, BetterFetchOption<any, any, any, User, {}, CustomHeaders>>(
  "https://api.example.com/users",
  {
    headers: {
      "x-tenant-id": "tenant-123",    // ✓ Type-safe
      "x-api-version": "v1"           // ✓ Type-safe
    }
  }
);
```

### Body

Request body types are inferred and validated:

```typescript
interface CreateUserDto {
  name: string;
  email: string;
  role: "admin" | "user";
}

const { data, error } = await betterFetch<User>(
  "https://api.example.com/users",
  {
    method: "POST",
    body: {
      name: "John Doe",
      email: "john@example.com",
      role: "admin"          // ✓ Type-safe enum
    } satisfies CreateUserDto
  }
);
```

### Query Parameters

Query parameters support type-safe definitions:

```typescript
interface UserQuery {
  page: number;
  limit: number;
  role?: "admin" | "user";
  verified?: boolean;
}

const query: UserQuery = {
  page: 1,
  limit: 10,
  role: "admin"
};

const { data } = await betterFetch<User[]>(
  "https://api.example.com/users",
  { query }
);
```

### URL Parameters

Dynamic URL parameters can be typed:

```typescript
interface UserParams {
  id: string;
  postId: string;
}

const params: UserParams = {
  id: "123",
  postId: "456"
};

const { data } = await betterFetch<Post>(
  "https://api.example.com/users/:id/posts/:postId",
  { params }
);
```

## Schema-Based Type Inference

For runtime validation and automatic type inference, use the `output` option with a schema library:

```typescript
import { z } from "zod";
import { betterFetch } from "@better-fetch/fetch";

const userSchema = z.object({
  id: z.string(),
  name: z.string(),
  email: z.string().email(),
  age: z.number().min(0),
  createdAt: z.string().datetime()
});

const { data, error } = await betterFetch(
  "https://api.example.com/users/123",
  {
    output: userSchema
  }
);

// TypeScript automatically infers the type from the schema!
if (data) {
  console.log(data.name);  // string
  console.log(data.age);   // number
  console.log(data.email); // string
}
```

<Note>
The `output` option works with any schema library that implements the [Standard Schema](https://github.com/standard-schema/standard-schema) specification, including Zod, Valibot, ArkType, and more.
</Note>

### Schema Validation Benefits

<Steps>
  <Step title="Automatic Type Inference">
    Types are inferred directly from your schema, keeping your types DRY:
    
    ```typescript
    import { v } from "valibot";
    
    const postSchema = v.object({
      id: v.number(),
      title: v.string(),
      published: v.boolean()
    });
    
    const { data } = await betterFetch(url, {
      output: postSchema
    });
    // data type is automatically: { id: number; title: string; published: boolean }
    ```
  </Step>
  
  <Step title="Runtime Validation">
    Your schema validates the response at runtime:
    
    ```typescript
    import { z } from "zod";
    
    const { data, error } = await betterFetch(url, {
      output: z.object({
        email: z.string().email() // Validates email format
      })
    });
    // If the API returns invalid data, it will be caught
    ```
  </Step>
  
  <Step title="Transform Data">
    Schemas can transform data during parsing:
    
    ```typescript
    import { z } from "zod";
    
    const { data } = await betterFetch(url, {
      output: z.object({
        createdAt: z.string().transform((str) => new Date(str)),
        price: z.string().transform((str) => parseFloat(str))
      })
    });
    
    if (data) {
      console.log(data.createdAt); // Date object
      console.log(data.price);     // number
    }
    ```
  </Step>
</Steps>

### Schema Validation Errors

When validation fails, a `ValidationError` is thrown:

```typescript
import { z } from "zod";
import { ValidationError } from "@better-fetch/fetch";

try {
  const { data } = await betterFetch(url, {
    output: z.object({
      email: z.string().email(),
      age: z.number().positive()
    })
  });
} catch (error) {
  if (error instanceof ValidationError) {
    console.log("Validation failed:", error.issues);
    // error.issues contains detailed validation errors
  }
}
```

## BetterFetchResponse Type

The return type changes based on the `throw` option:

### Default (Error as Value)

```typescript
type BetterFetchResponse<T, E, Throw = false> = 
  | { data: T; error: null }
  | { data: null; error: E & { status: number; statusText: string } };

const result = await betterFetch<User>(url);
// result is: { data: User; error: null } | { data: null; error: { status: number; statusText: string } }
```

### Throw Mode

```typescript
const result = await betterFetch<User>(url, { throw: true });
// result is: User (no wrapper object)
// Errors are thrown as BetterFetchError
```

## Type-Safe Client with createFetch

Create a fully typed client using `createFetch`:

```typescript
import { createFetch } from "@better-fetch/fetch";

interface ApiConfig {
  baseURL: string;
  headers: {
    "x-api-key": string;
  };
}

const api = createFetch({
  baseURL: "https://api.example.com",
  headers: {
    "x-api-key": process.env.API_KEY
  },
  throw: true
} satisfies ApiConfig);

// All requests inherit the configuration and throw mode
interface User {
  id: string;
  name: string;
}

try {
  const users = await api<User[]>("/users");
  // users is User[] directly (no wrapper)
  console.log(users[0].name);
} catch (error) {
  // Errors are thrown
  console.error(error);
}
```

## Advanced Type Patterns

### Discriminated Unions

Handle multiple response types with discriminated unions:

```typescript
type ApiResponse<T> = 
  | { success: true; data: T }
  | { success: false; error: string };

interface User {
  id: string;
  name: string;
}

const { data } = await betterFetch<ApiResponse<User>>(
  "https://api.example.com/users/123"
);

if (data && data.success) {
  // TypeScript knows data.data is User
  console.log(data.data.name);
} else if (data && !data.success) {
  // TypeScript knows data.error is string
  console.error(data.error);
}
```

### Generic API Functions

Create reusable generic functions:

```typescript
async function getById<T>(endpoint: string, id: string): Promise<T | null> {
  const { data, error } = await betterFetch<T>(
    `https://api.example.com/${endpoint}/${id}`
  );
  
  if (error) {
    console.error(`Failed to fetch ${endpoint}:`, error);
    return null;
  }
  
  return data;
}

interface User {
  id: string;
  name: string;
}

interface Post {
  id: number;
  title: string;
}

// Type is inferred from generic parameter
const user = await getById<User>("users", "123");
const post = await getById<Post>("posts", "456");
```

### Conditional Types

Use conditional types for flexible APIs:

```typescript
type FetchOptions<ThrowMode extends boolean> = {
  throw: ThrowMode;
};

type FetchResult<T, ThrowMode extends boolean> = 
  ThrowMode extends true 
    ? T 
    : { data: T | null; error: Error | null };

async function safeFetch<T, ThrowMode extends boolean = false>(
  url: string,
  options?: FetchOptions<ThrowMode>
): Promise<FetchResult<T, ThrowMode>> {
  return betterFetch<T>(url, options as any) as any;
}

// Infers throw mode from options
const result1 = await safeFetch<User>(url, { throw: true });
// result1 is User

const result2 = await safeFetch<User>(url);
// result2 is { data: User | null; error: Error | null }
```

## Type Utilities

Better Fetch exports useful type utilities:

```typescript
import type {
  BetterFetchOption,
  BetterFetchResponse,
  CustomHeaders,
  Method,
  PayloadMethod,
  NonPayloadMethod
} from "@better-fetch/fetch";

// Use in your own types
type MyRequestOptions = BetterFetchOption<
  { name: string },  // Body type
  { page: number },  // Query type
  { id: string }     // Params type
>;

type MyHeaders = CustomHeaders<{
  "x-tenant-id": string;
}>;

type GetMethod = NonPayloadMethod; // "GET" | "HEAD" | "OPTIONS"
type PostMethod = PayloadMethod;   // "GET" | "POST" | "PUT" | "PATCH" | "DELETE"
```

## Best Practices

<Steps>
  <Step title="Define Interfaces for API Types">
    Create TypeScript interfaces for your API models:
    
    ```typescript
    // api-types.ts
    export interface User {
      id: string;
      name: string;
      email: string;
    }
    
    export interface ApiError {
      code: string;
      message: string;
    }
    ```
  </Step>
  
  <Step title="Use Schema Libraries">
    Prefer schema-based validation for runtime safety:
    
    ```typescript
    import { z } from "zod";
    
    // Schema doubles as type definition and validator
    export const userSchema = z.object({
      id: z.string(),
      name: z.string(),
      email: z.string().email()
    });
    
    export type User = z.infer<typeof userSchema>;
    ```
  </Step>
  
  <Step title="Type Your Error Responses">
    Always specify error types for better error handling:
    
    ```typescript
    const { data, error } = await betterFetch<User, ApiError>(url);
    
    if (error) {
      // TypeScript knows error structure
      console.log(error.code);
    }
    ```
  </Step>
  
  <Step title="Create Typed API Clients">
    Encapsulate API calls in typed client classes or functions:
    
    ```typescript
    class UserApi {
      private client = createFetch({
        baseURL: "https://api.example.com"
      });
      
      async getUser(id: string): Promise<User | null> {
        const { data, error } = await this.client<User>(`/users/${id}`);
        return error ? null : data;
      }
      
      async createUser(dto: CreateUserDto): Promise<User | null> {
        const { data, error } = await this.client<User>("/users", {
          method: "POST",
          body: dto
        });
        return error ? null : data;
      }
    }
    ```
  </Step>
</Steps>
